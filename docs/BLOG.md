# ChatGPT 是怎样炼成的

GPT 的全称是 **Generative Pre-trained Transformer**，生成式预训练 Transformer。

Transformer 原意是 "变压器"。在电路里，变压器会把一种电压转换成另一种电压；在语言模型里，Transformer 做的也是一种 "转换"：它把一串原始 token 表示，一层层转换成带有上下文理解的向量表示。

一开始，每个 token 只知道自己是谁。比如 "它" 这个 token，单独看只是一个代词；但经过 Transformer 之后，它会根据前面的线索，不断更新自己的表示。最后模型看到的就不再是孤立的 "它"，而是一个已经吸收了上下文信息的 "它"。

要制作一个属于自己的 GPT，我们需要从最基础的 token 开始讲起。

## Tokenizer：模型看不到文字，它只看到数字

假设我们面前有一个幼儿园小朋友。我们想让她读懂《战争与和平》，甚至以后还能自己写同人续作。

问题来了：她一开始既不认识 "词"，也不认识 "字"，更不知道代码里的 for、return、async、await 是什么。她唯一会做的事情，是掰手指头数数。

这其实就是机器刚开始面对文本时的处境。模型并不是直接看原始文字的，它真正能处理的是数字序列，也就是 token id。

所以在模型开始回答 "下一个词是什么" 之前，我们必须先解决一个更基础的问题：

**怎样把一整本书、一段代码、一句话，拆成模型能数、能记、能学习的小块？**

这个 "分词器" 或者说 "切块器"，就是 Tokenizer。

### Tokenizer 到底在干什么？

如果我们是幼儿园老师，我们得先决定：

1. 是按单个字符教她？
2. 还是按常见词教她？
3. 还是按更灵活的 "常见片段" 教她？

不同的切法，会直接影响后面学习的难度。

举个最简单的英文例子：

```text
the cat sat on the chair
```

如果切得很碎，可能会变成：

```text
t | h | e |   | c | a | t | ...
```

这样模型要处理很多步。

如果切得更合理，可能会变成：

```text
the | cat | sat | on | the | chair
```

这样序列就短很多。

再换一个代码例子：

```python
for i in range(10):
```

如果 Tokenizer 不熟悉代码，它可能切成很多碎片：

```text
f | o | r |   | i |   | i | n |   | r | a | n | g | e | ( | 1 | 0 | ) | :
```

但如果它见过很多代码，可能会更倾向于形成像下面这样的片段：

```text
for |  i |  in |  range | ( | 10 | ):
```

这里没有唯一标准答案，但预期很清晰：

**高频片段如果能作为整体出现，模型会更省力。**

### 为什么要自己训练 Tokenizer？

现成 Tokenizer 很多，比如 GPT-2 的 Tokenizer、`cl100k_base`、Llama 系 Tokenizer。

很多时候直接拿来用完全没问题，但 Tokenizer 有一个很重要的特点：

**它不是 "通用越强越好"，而是 "越贴合语料越好"。**

比如：

- 如果我们的数据大部分是英文新闻，英文常见词切得好就很重要；
- 如果我们的数据里代码很多，那 `def`、`return`、`http`、`</div>` 这类片段能不能高效表示就很重要；
- 如果我们的数据里中文很多，那常见中文词组能不能不要被切得太碎就很重要；
- 如果我们的数据里数学公式很多，那 `\frac`、`\sum`、`^{2}` 这类片段能不能成块出现也很重要。

所以自己训练 Tokenizer，本质上是在做一件事：

**让数据里最常见的模式尽量以更合适的颗粒度进入模型。**

### BPE (Byte Pair Encoding)：让高频片段慢慢长出来

GPT 的 Tokenizer 训练，本质上是 BPE (Byte Pair Encoding) 的一个变体。

BPE (Byte Pair Encoding) 最好理解的方式，不是先看定义，而是直接看它 "怎么生长"。

假设训练文本里有很多下面这样的词：

```text
the
this
that
there
```

一开始，系统并不认识 `the` 这个整体。

它只认识最基础的单位，然后训练过程会不断统计：

**哪些相邻片段经常一起出现？**

于是发生这样的合并：

#### 第一步

发现 `t` 和 `h` 经常相邻出现，于是合成：

```text
t + h -> th
```

#### 第二步

又发现 `th` 和 `e` 很常一起出现，于是继续合成：

```text
th + e -> the
```

#### 第三步

如果某些更长片段也足够常见，还可能继续合并。

比如在代码里，假设训练语料中到处都是：

```text
return
```

那么它也可能经历类似过程：

```text
r + e -> re
re + t -> ret
ret + urn -> return
```

实际内部过程当然会更复杂，不一定正好按这个顺序，但直觉就是这个直觉：

**BPE 不是人手工规定哪些词最重要，而是让高频片段自己从数据里长出来。**

### 为什么不是直接按 "单词" 切

既然最终目标是让文本别切太碎，那为什么不干脆按空格分词，或者按单词分词？

原因是现实文本远比英语单词列表复杂。

看几个例子：

#### 例子 1：新词和罕见词

```text
hyperparameterization
```

如果词表里没有这个词，按整词分就会尴尬。但 BPE (Byte Pair Encoding) 可以退回到更小片段，比如：

```text
hyper | parameter | ization
```

哪怕还不行，也能继续退回到更小单位。

#### 例子 2：代码和符号

这里既有字母，也有下划线、等号、数字。按自然语言单词思路就不太好处理。

```text
user_id=42
```

#### 例子 3：中文

中文本来就没有天然空格分词。 "机器学习工程师" 到底该切成：

```text
机器 | 学习 | 工程师
```

还是：

```text
机器学习 | 工程师
```

甚至：

```text
机 | 器 | 学 | 习 | 工 | 程 | 师
```

并没有一个放之四海而皆准的固定答案。

BPE (Byte Pair Encoding)  的好处就是：它让数据自己决定哪些组合值得变成更稳定的 token。

### Tokenizer 的产出物，到底是什么

如果我们亲手训练一个 Tokenizer，最后得到的并不是 "一段文本的切分结果"，而是一套以后可以反复复用的规则。

最核心的两个产物可以这样理解：

#### 1. `Tokenizer.pkl`

这是分词器本体。它里面包含的是：

- token 和 id 的映射
- merge 规则
- special token 配置

类似这样的结构：

```json
{
   "token_to_id": {
      "the": 101,
      "你": 102,
      "return": 103,
      "<|bos|>": 0
   },
   "id_to_token": {
      "101": "the",
      "102": "你",
      "103": "return",
      "0": "<|bos|>"
   },
   "merges": [
      ["t", "h", "th"],
      ["th", "e", "the"],
      ["ret", "urn", "return"]
   ],
   "special_tokens": {
      "bos": "<|bos|>",
      "user_start": "<|user_start|>",
      "assistant_start": "<|assistant_start|>"
   }
}
```

真实的 pkl 内部格式未必长得一模一样，但核心意思就是：给定文本片段，能查到它的 id；给定 id，也能还原对应 token；遇到新文本时，再按 merge 规则一步步切分。

也就是说，它决定了：

**以后所有文本，到底该怎么被切成 token。**

#### 2. `token_bytes.pt`

这是另一张配套表：

```text
token_id -> 这个 token 对应多少个 UTF-8 字节
```

举个例子。假设某个 Tokenizer 里有这些 token：

```text
token 101 -> "the"
token 102 -> "你"
token 103 -> "return"
```

那么它们对应的 UTF-8 字节数是：

```text
"the"     -> 3 bytes
"你"      -> 3 bytes
"return"  -> 6 bytes
```

注意这里记录的是字节数，不是字符数，token 和字节数的映射关系是后面 BPB 评估的基础。

### BPB (Bits per Byte)：为什么不能只按 token 计分

假设同一句话：

```text
I love deep learning
```

被两个 Tokenizer 分成这样：

#### Tokenizer A

```text
I | love | deep | learning
```

一共 4 个 token。

#### Tokenizer B

```text
I |  l | ove |  de | ep |  learn | ing
```

一共 7 个 token。

这时如果只按 token 个数来比较，就不太公平。因为两个 Tokenizer 的颗粒度不一样，同样一句话，在不同 Tokenizer 眼里会变成不同数量的 token。

更稳的比较方式是：

**为了描述同样一段原始文本，模型平均每个 byte 需要付出多少信息量？**

这就是 BPB (Bits per Byte) 的想法。

它就像物流计费：

- 按箱收费，不公平，因为箱子大小不同；
- 按公斤收费，更统一。

同理：

- 按每 token 算，不同 Tokenizer 下 "箱子大小" 不一样；
- 按每 byte 算，就更接近统一度量尺度。

> 某种意义上，Tokenizer 做的就是模型世界里的 "书同文，车同轨"：
>
> 先把不同语种、代码、数学公式这些形态各异的文本，统一转换成 token id 这条共同轨道。只有先进入同一套表示系统，后面的 Embedding、Attention 和训练过程，才有可能在它们之间建立联系。


## Embedding：将 Token 变成高维向量

Tokenizer 把原始文本翻译成了模型能处理的 token id。

比如一句话：

```text
I love deep learning
```

经过 Tokenizer 之后，会变成：

```text
[40, 1842, 6732, 12930]
```

**但 token id 只是一个编号。**

编号只能告诉模型：这是词表里的第几号 token。它本身并不能表达更丰富的含义：比如 "猫" 和 "狗" 更接近，"猫" 和 "函数" 更远；"return" 常出现在代码里，"苹果" 常出现在日常语言里。

所以接下来的一步，是给每个 token id 分配一个可以学习的向量。

把 token id 映射成向量的过程，叫做 Embedding，得到一个 Embedding Table，可以把 Embedding Table 想成一本超大的 "语义坐标表"，每个 token 都会被放到一个长度为 `n_embd` 的向量里：

```text
40    -> [0.12, -0.33, 0.88, ...]
1842  -> [-0.51, 0.27, 0.04, ...]
6732  -> [0.91, -0.10, 0.22, ...]
```

### 初始 Embedding Table 是怎么来的？

在一开始，模型并不知道 "猫" "狗" "苹果" "函数" "变量" 是什么意思。

Embedding Table 通常是随机初始化的。

假设要学习的语料被分词器切割成 50000 个 token，那么 Embedding Table 的大小 `vocab_size` 也是 50000：

```
vocab_size = 50000 (token)
```

向量的维度 `n_embd` 是：

```
n_embd = 4096
```

那 Embedding Table 就是一个 50000 行 × 4096 列 的矩阵：

```
E ∈ R^(50000 × 4096)
```

每一行对应一个 token 的向量，也就是说，每个 token 都有一个 4096 维的可学习向量。

```
token_id = 40    →  [0.003, -0.012, 0.008, ...]
token_id = 1842  →  [-0.010, 0.004, 0.017, ...]
token_id = 6732  →  [0.006, 0.011, -0.002, ...]
```

训练开始时，这些数字一开始没有语义，只是很小的非 0 随机数。

通过持续的训练，不断更新，Embedding Table 才变得有意义。

### Embedding 如何学出语义？

比如训练文本里经常出现：

```
猫 喜欢 吃 鱼
狗 喜欢 吃 骨头
猫 和 狗 都 是 动物
```

模型在预测时会遇到很多类似任务：

```
猫 喜欢 吃 ___
```

如果正确答案经常是 "鱼"，模型就会调整参数，让 "猫" 这个 token 的向量更有利于预测 "鱼" "动物" "宠物" 等相关 token。

这是通过大量预测任务训练出来的，所以 Embedding 的语义来自训练语料中的共现关系。

### Embedding 维度大小怎么决定？

向量的维度 `n_embd`，其实就是模型内部主干向量的维度 `d_model`，是模型设计者设定的参数（也叫超参数，即预先定义好的常量）：

```
小模型：d_model = 512 / 768 / 1024
中模型：d_model = 2048 / 3072 / 4096
大模型：d_model = 5120 / 8192 / 12288 ...
```

维度越高，表达空间越宽，单个 token 能携带的信息容量越大，但代价也越高：

- 模型参数量增加
- 显存增加
- 计算量增加
- 训练更贵
- 推理更慢

维度大小通常会和这些东西一起预先设计好：

- Transformer Block 的层数/深度：depth
- 注意力头数：num_heads
- MLP 中间层维度
- 训练 token 数
- 算力预算
- 目标模型规模

现在我们已经知道：token 会先被 Tokenizer 变成 token id，再被 Embedding 变成向量。接下来的问题是：这些向量怎样互相看见彼此，怎样把上下文信息融合进自己的表示里？

这就进入了最核心的部分：Attention。

## Attention Is All You Need

### 注意力机制是什么？

OpenAI 早期 GPT 论文的核心思路，是先在大规模未标注文本上进行语言建模式预训练，再迁移到具体任务上。它使用的 Transformer 架构来自 2017 年的 *Attention Is All You Need*。

注意力机制解决的问题是：

> 当前 token 在理解自己时，应该重点参考前面的哪些 token？

比如句子：

```
小明把苹果放进书包，因为它很重。
```

"它" 到底指谁？模型需要看上下文。Attention 就是在计算：

```
当前 token 和其他 token 的相关性
```

核心公式是：
$$
\text{Attention}(Q, K, V) = \text{softmax}\left(\frac{QK^T}{\sqrt{d_k}}\right) V
$$
其中：

```
Q = Query：我现在想找什么？
K = Key：我这里有什么信息可以被别人匹配？
V = Value：如果别人关注我，我实际提供什么内容？
```

就好比相亲，每个 token 就是场上的一个人，胸前同时挂着三块牌子：

```
Q（择偶要求）："我在找一个会做饭的人"
K（自我标签）："我会做饭、我会编程、性格开朗"
V（我真实的详细内容）：一本写满你完整故事和能力的履历
```

匹配过程：

1. 拿自己的 **Q**（要求）去看场上每个人的 **K**（标签），算相似度
2. 找到跟自己的 **Q**（要求）最匹配的那几个人，重点听他们的 **V**（详细内容）
3. 把听到的内容融合，更新自己对当前局势的理解

为什么要拆成 Q、K、V 三块？

因为每个 token 在注意力中需要 **同时扮演两种角色**：它既要 "问别人"（用 Q 去匹配别人的 K），也要 "被别人问"（用 K 被匹配，被选中后贡献 V）。所以一个 token 必须同时具备：能发出问题的形态（Q）+ 能被搜索到的标签（K）。

而 K 和 V 又必须分开，是因为 **"能被搜索到的标签" 和 "真正能提供的内容" 不是一回事**

也就是说：

```
同一个 token 向量 x
 → x · Wq = q：变成 "提问者" 形态，问：我现在想找什么？
 → x · Wk = k：变成 "被搜索者" 形态，亮出：我有什么标签？
 → x · Wv = v：变成 "内容提供者" 形态，准备：被选中我就交出这些内容
```

三种角色都源于同一个语义（这个词的本质含义），但用途不同 —— 就像同一个人写简历找工作 vs 写自我介绍交友 vs 跟朋友深聊，核心是同一个个体，但呈现的 "切面" 不一样。

### Linear Layer 线性层：换一个角度看 token

上面公式里出现的 W_Q、W_K、W_V，它们都 是**矩阵**，大小是 `d_model × d_model`，里面装的就是模型的参数。

```
Wq ∈ R^(d_model × d_model)
Wk ∈ R^(d_model × d_model)
Wv ∈ R^(d_model × d_model)
```

把一个向量乘以这样一个矩阵，就是一次 **线性变换（Linear Transformation）**，神经网络里负责做这件事的模块就叫 **线性层（Linear Layer）**，PyTorch 里就是 `nn.Linear`。

直觉上可以把线性层理解成：**一台架在某个角度的摄像机**。同一个 token 向量就像一个立体的物体，从不同角度拍它，画面里看到的内容是不一样的：

- 用 Wq 这台摄像机拍  →  拍到的是 "它想找什么"
- 用 Wk 这台摄像机拍  →  拍到的是 "它能被什么标签匹配"
- 用 Wv 这台摄像机拍  →  拍到的是 "它能贡献什么内容"

这三台摄像机的架设角度（即矩阵里的具体数值）不是人手工设定的，而是在训练中由数据自己优化出来的 —— 模型会摸索出：到底从哪个角度拍一个 token，才能让 Attention 算得最准。

注意：这里的 "三台摄像机" 是因为我们需要把 token 拆成 **Q、K、V 三种角色**，每种角色对应一台摄像机（一个线性层），少一台都不行。

这就是为什么 GPT 的模型中存在三个线性层。

但还有另一个问题：**每台摄像机自己只能从一个角度拍**。也就是说，W_Q 这台摄像机只能用一个固定的 "提问视角" 去问问题。可现实中，一个 token 想问的问题往往不止一个 —— 它既想问 "谁跟我语义相关"，也想问 "谁是我的语法搭档"，还想问 "谁是我指代的对象"。

那能不能让 Q 这一类摄像机有多个，每台从不同角度提问？K 和 V 也一样？这就是 **多头注意力**。

### 多头注意力：每种角色都从多个角度看

现实中我们去相亲，关心的肯定不止 "会不会做饭" 这一个维度。我们同时还关心性格、兴趣、收入、家庭、价值观……如果只用一个 Q 把所有需求糅在一起去匹配，信息会被压缩成 "综合评分"，很多细节就丢失了。

更聪明的做法是：**同时派出多个分身去匹配**，每个分身只关心一个维度。

```
一个 token → 拆成 8 个分身，每个分身有自己的 Q/K/V：
 分身 1：Q_1 找性格合拍的     → 看每个人的 K_1 性格标签 → 取 V_1
 分身 2：Q_2 找兴趣相近的     → 看每个人的 K_2 兴趣标签 → 取 V_2
 分身 3：Q_3 找指代关系的     → 看每个人的 K_3 指代标签 → 取 V_3
 ...
 分身 8：Q_8 找语法搭配的     → 看每个人的 K_8 语法标签 → 取 V_8
```

每个分身独立完成一次 Attention，得到一份针对某个维度的 "参考结果"，最后把 8 份结果拼接起来，就是这个 token 的最终输出。

数学上做的其实很简单：把 `d_model` 维的向量切成 `num_heads` 个子空间，每个子空间维度是 `d_head = d_model / num_heads`，每个 head 在自己的子空间里独立算一次 Attention，最后拼回原维度。

```
d_model = 512, num_heads = 8  →  每个 head 在 64 维子空间里工作
```

为什么要这么做？

- **单头容易把不同关系混在一起**。比如一个 head 既要学指代关系（"它" 指 "苹果"），又要学语法关系（"放进" 的宾语是 "书包"），互相干扰。多头让每个 head 可以专注一种 "关注模式"。
- **表达能力更丰富**。8 个 head 相当于从 8 个不同的角度看同一段上下文，捕捉到的关系组合远比单头多。
- **几乎不增加计算量**。因为每个 head 维度变小了（64 而不是 512），8 个 head 加起来的总计算量和单头 512 维差不多，但表达力强很多。

更有意思的是：这些 "切面" 不是人手工设计的，而是**模型自己在训练中学出来的**。训练好的多头注意力里，可能确实有些 head 专门关注指代、有些关注句法依赖、有些关注相邻位置、有些关注全局主题 —— 这完全是数据驱动的角色分工。


## Transformer Block：一个 token 的完整旅程

### 只会听别人说话，还不够

多头注意力让一个 token 可以从很多角度去看上下文。

比如处理这句话里的 "它"：

```text
小明 把 苹果 放进 书包 因为 它 很 重
```

某个 head 可能更关注 "它" 和 "苹果" 的指代关系，另一个 head 可能更关注 "很重" 这个属性，还有的 head 可能在看 "放进" 前后的语法结构。

这很像开三方会议。Attention 做的是：让每个 token 去听一圈别人怎么说，把跟自己有关的信息带回来。

但问题是，**听完别人说话，不等于已经想明白了。**

"它" 听到了 "苹果"，也听到了 "书包"，还听到了 "很重"。接下来它还得在自己的位置上继续消化：

```text
"很重" 更像是在说苹果，还是书包？
"因为" 后面的内容是在解释前面的哪个动作？
如果要预测下一个 token，现在这些线索应该怎么组合？
```

所以在一个 Transformer block 里，Attention 后面通常还会接一个小的神经网络（**MLP / FFN**），专门负责让每个 token 在自己的位置上继续处理信息。

### MLP / FFN：每个 token 的独立加工

MLP 的全称是 **Multi-Layer Perceptron（多层感知机）**。在 Transformer 里，它也常被叫做 **FFN（Feed-Forward Network，前馈神经网络）**。

这个名字听起来有点大，其实先不用怕。它在这里做的事情，可以从刚才的会议比喻继续理解。

Attention 像开会：每个 token 都可以听别人发言。可会议本身只负责把信息带到桌面上，不负责替每个人写结论。

散会之后，每个 token 都要回到自己的工位，把刚才听到的信息重新整理成自己的判断。

比如 "它" 这个位置拿到了几条线索：

```text
苹果：可能是被指代对象
书包：也可能是被指代对象
很重：更像是在描述某个物体的属性
```

这些线索不能简单相加。模型需要学会一些更复杂的组合方式：

```text
如果 "很重" 更常修饰苹果，那就提高苹果相关判断的权重；
如果上下文说的是 "放进书包"，那书包也不能完全排除；
如果后面要预测下一个 token，还要把语法位置也考虑进去。
```

这种 "把已有线索重新组合、筛选、变形" 的活，就交给 MLP。

在 MLP 里 token 之间不会再相互交流了，Attention "会议" 已经完成了跨 token 的信息交换，到了 MLP 这里，每个 token 在自己的工位上自闭，用同一套方法论来处理线索。

"它" 这个 token 要整理自己的线索，"苹果" 这个 token 也要整理自己的线索，"书包" 这个 token 也一样。

**Q、K、V 只是在 Attention 里面临时使用的三种角色。** 一个 token 先从自己的向量变出 Q、K、V，用它们完成一轮 "提问、匹配、取信息"。Attention 算完之后，这三种临时角色就已经完成任务了，留下来的不是 Q、K、V，而是这个 token 更新后的表示。

这个 "更新后的表示"，就是模型学习的结果之一。它不是人手工写规则算出来的，而是由一堆可学习的参数共同决定的：前面 Embedding 词表里的每一行也是可学习的参数，生成 Q/K/V 的三个矩阵也是可学习的参数。

训练时，模型不断预测下一个 token。预测错了，就会调整这些参数：Embedding 词表会被更新，W_Q / W_K / W_V 会被更新。久而久之，同一个 token 经过这些参数加工后，就会得到越来越有用的向量表示。

### MLP 的结构

MLP 常见写法是 "两个线性层 + 激活函数"

```
x → Linear(d_model → d_ff) → 激活函数 → Linear(d_ff → d_model)
```

我们可以拆开看。

第一层线性层的输入，是 Attention 之后某个 token 的当前表示：

MLP 接到的输入不是 QKV，而是经过 Attention 更新之后的那个 token 向量 `x`：

它仍然是 `d_model` 维，只是里面已经带上了刚才从上下文听回来的信息。

```text
x ∈ R^d_model = 某个 token 在当前 block 里的表示
```

比如还是 "它" 这个位置。经过 Attention 之后，这个向量里已经混进了上下文线索：苹果、书包、很重、因为、放进这些信息都以某种方式压在同一个 `d_model` 维向量里。

第一层线性层会把它投到一个更宽的空间 `d_ff`：

```
d_model = 512
d_ff    = 2048
```

这 2048 个中间维度不是凭空冒出来的新信息。它们更像是 2048 个模型自己学出来的 "候选特征探测器"。每个探测器都从原来的 512 维里看一组不同的组合：

```text
第 17 个中间维度：也许在看 "它 + 很重" 这类线索
第 231 个中间维度：也许在看 "放进 + 书包" 这类线索
第 890 个中间维度：也许在看 "某个物体属性是否正在被解释" 这类线索
```

这些解释只是帮助理解，真实模型不会给每个维度贴中文标签。但直觉是对的：升维不是把信息变多，而是把原来挤在一起的信息摊开，让模型有更多中间位置去表示各种组合线索。

但如果只有线性层，事情还不够有趣。因为两个线性层连在一起，本质上还是一个线性层：

```
Linear(Linear(x))  仍然等价于  Linear(x)
```

沿用前面 "摄像机" 的类比，线性层就像换一个拍摄角度。连续换两次角度，仍然可以合并成一次更复杂的拍摄角度。中间如果没有别的东西，模型只是把画面从一个坐标系搬到另一个坐标系。

可语言里的很多关系不是直线式的。

比如 "很重" 单独出现时，不一定说明什么；"书包" 单独出现时，也不一定说明什么；但当 "苹果 放进 书包 因为 它 很 重" 这些线索同时出现时，模型需要突然更相信 "它" 指的是苹果。

这类逻辑更像是：某几个特征同时亮起来之后，另一个判断才被点亮。只靠连续换拍摄角度，很难产生这种 "亮到某个程度才触发" 的效果。

所以中间必须插入一个 **激活函数**。

激活函数可以理解成接在 2048 个候选特征后面的 "调光开关"：有些特征很弱，就被压下去；有些特征很强，就被放过去；有些特征处在中间地带，就被平滑地削弱或保留一部分。

这样一来，中间层就不只是 "换角度拍照"，而是能产生真正的非线性效果：

```text
某个组合线索不够强 → 不触发
某几个线索一起出现 → 触发得更明显
某个线索方向相反 → 被压低
```

这就是 MLP 能表达复杂关系的关键。它不是只把 512 维搬到 2048 维再搬回来，而是在更宽的中间空间里，让一些组合特征亮起来，让另一些组合特征暗下去。

Transformer 里常见的激活函数有 ReLU / GELU / SiLU。

- **ReLU** 像硬开关：大于 0 就放行，小于 0 就关掉。它简单、便宜、有效，但开关太硬，负半边的信息会直接变成 0。
- **GELU** 像更细腻的调光器：信号越强，越倾向于通过；信号越弱，越倾向于被压低，但不是一刀切。GPT-2、BERT 等模型里经常能看到它。
- **SiLU** 也像调光器，不过它更像 "输入自己控制自己的门"：信号越合适，门开得越大。很多现代 LLM 的 MLP 变体会使用 SiLU 或类似的门控思路。

为什么会有不同选择？因为这里其实是在取舍：硬开关计算简单，平滑开关训练更柔和，门控式开关更适合某些现代 MLP 结构。它们的具体曲线不同，但目标都是同一件事：**让 MLP 不只是换坐标，而是真的能根据输入内容决定哪些特征该亮、哪些特征该暗。**

经过激活函数之后，我们得到的是一个 `d_ff` 维的中间表示。它不是下一个 block 能直接接收的格式，因为整个 Transformer block 的输入输出宽度都约定为 `d_model`。

所以第二层线性层要把它压回 `d_model` 维：

```
d_model → d_ff → d_model
```

这一步不是简单地 "删掉多余维度"。更准确地说，它会把刚才 2048 个候选特征重新混合，压缩成一个 512 维的输出向量。这个输出仍然对应同一个 token 位置，但它已经带上了 MLP 加工后的结果：哪些线索更重要，哪些组合关系更值得保留，都会被写回 `d_model` 维的表示里。

当然，这个过程通常不是可逆的。2048 维中间状态不会被完整保存下来，模型只保留对下一层有用的压缩结果。损失什么、保留什么，也不是人手工规定的，而是在训练中通过预测下一个 token 这件事学出来的。

$$
\mathrm{MLP}(x) = W_2 \cdot \text{GELU}(W_1 \cdot x + b_1) + b_2
$$

这条公式展开后，就是刚才那条流水线：

```
升维 → 非线性加工 → 降维
```

更具体一点：

```text
输入：一个 token 当前的 d_model 维表示
升维：展开成 d_ff 个候选特征
激活：让该亮的特征亮起来，该压低的特征暗下去
降维：把加工结果写回 d_model 维，交给后面的 block
```

一个 token 在 Transformer block 里都要经历：

```
Attention（开会，和上下文交换信息）
   ↓
MLP（在当前位置内部加工信息）
```

这两步合起来，才像一次完整的 "理解更新"：先听别人，再自己想一想。

为什么需要很多个 Transformer block 连在一起？

因为一次更新通常只能完成一轮比较浅的整理。第一层可能先看清局部搭配，第二层在这个基础上看语法关系，更高的层再继续处理指代、语义和预测下一个 token 需要的线索。每一层都接收上一层整理好的表示，再做一轮新的 Attention 和 MLP。

所以多层不是简单重复，而是让 token 的表示一层层被改写：

```text
原始 token 表示
→ 加入一轮上下文线索
→ 再整理一轮组合关系
→ 再看更高层的语义线索
→ 最后用于预测下一个 token
```

token 先变成向量，经过一层层 Transformer block，最后得到一个更成熟的 token 表示。

### 从表示到预测：logits 和 softmax

经过很多层 Transformer block 之后，位置 `i` 的 token 得到一个最终向量：

```text
h_i ∈ R^d_model
```

这个向量已经融合了前文信息，也经过了多轮加工。但它本身还不是答案。模型最终要回答的是：

```text
下一个 token 是 vocab 里的哪一个？
```

vocab 通常有几万个 token，所以模型需要把 `d_model` 维向量变成一个 `vocab_size` 维的打分向量。这个打分向量就叫 **logits**：

```text
logits_i = h_i · W_lm_head
```

这里的 `W_lm_head` 也叫 **lm_head（Language Modeling Head）**，本质上还是一个线性层：它负责把模型内部的 `d_model` 维表示，投到整个词表大小的空间里。

如果 vocab 里有 50000 个 token，那 `logits_i` 就有 50000 个数。每个数对应一个候选 token 的分数：

```text
"苹果"：  3.2
"书包"：  1.1
"很"：   -0.7
"。"：    2.4
...
```

这些分数还不是概率。它们可以是负数，也不要求加起来等于 1。要把它们变成概率分布，就需要再过一次 **softmax**：

```text
P(下一个 token | 上文) = softmax(logits_i)
```

softmax 做的事情可以粗略理解成：把所有候选 token 的分数放在一起比较，分数高的概率变大，分数低的概率变小，最后所有概率加起来等于 1。

于是模型就从一个 `d_model` 维的内部表示，走到了一个可以读懂的预测结果：

```text
token 向量 → 多层 Transformer block → logits → softmax → 下一个 token 的概率
```

还有一个常见技巧叫 **weight tying**：让 lm_head 复用 token Embedding 矩阵的转置。直觉上，输入端的 Embedding 在学 "每个 token 应该是什么向量"，输出端的 lm_head 在学 "哪个向量更像某个 token"。这两件事天然有关联，所以很多 GPT 实现会把这两组参数绑在一起，既省参数，效果通常也不错。

不过 token Embedding 和 lm_head 也可以是两套独立参数。也就是说，输入端有一张 "token → 向量" 的表，输出端还有一张独立的 "向量 → token 打分" 的表。这样参数会多一点，但也给模型多了一点自由度。

### 从预测错误到参数更新

训练时，我们知道真实的下一个 token 是什么。

比如输入是：

```text
小明 把 苹果 放进 书包
```

某个位置的正确答案可能是：

```text
因为
```

如果模型给 "因为" 的概率很低，就说明它这次预测得不好。我们需要一个数字来衡量“错得有多离谱”，这就是 **loss（损失）**。

语言模型里最常用的是 **交叉熵损失（Cross-Entropy Loss）**。直觉很简单：

```text
正确答案概率越高 → loss 越小
正确答案概率越低 → loss 越大
```

写成公式就是：

```text
loss_i = -log P(真实下一个 token | 上文)
```

然后训练会做一件关键的事：根据这个 loss，反过来调整模型里要学习的参数。

这些参数包括前面提到过的很多东西：

- Embedding 词表里的 token 向量
- Attention 里的 W_Q、W_K、W_V、W_O
- MLP 里的 W_1、W_2
- 最后把 hidden state 变成 logits 的 W_lm_head

这个 "根据 loss 反过来计算每个参数该怎么改" 的过程，就是 **反向传播（backpropagation）**。真正执行参数更新的工具，通常叫 **优化器（optimizer）**，比如现代 LLM 训练里很常见的 AdamW、最新的 Muon。

所以一次训练可以先粗略理解成：

```text
正向传播：token → Embedding → Transformer blocks → logits → softmax → loss
反向传播：根据 loss 计算每个参数该往哪边改
优化器：按照这些方向，更新 Embedding、Attention、MLP、lm_head 等参数
```

这也解释了前面说的：Embedding 词表不是固定死的。它一开始只是随机向量，但每次预测错了，训练都会把错误信号传回去，Embedding 词表也会跟着一起更新。久而久之，"苹果" "书包" "很重" 这些 token 的向量才会越来越有语义。

到这里，GPT 的训练主流程已经串起来了：模型怎样读入 token，怎样一层层加工表示，怎样输出预测，又怎样根据错误更新参数。

---

前面为了先把主线讲清楚，我们把 Transformer block 画得比较粗：

```text
Attention（和上下文交换信息）
   ↓
MLP（在当前位置内部加工信息）
```

这张图还少了一个很关键的细节：Attention 和 MLP 算完之后，结果到底怎么接回原来的 token 表示里？

最直接的想法是让 Attention 的输出直接变成新的表示，再把这个新表示交给 MLP：

```text
x₀ → Attention → MLP → x₁
```

但这样一来，每个子层都像是在说："前面的表示不用了，我重新写一份。"

模型只有一两层时，这件事似乎还不明显。可是 GPT 往往要堆很多层。如果每一层都把上一层的表示整份改写掉，前面好不容易学到的局部搭配、语法线索、指代线索，就可能在后面的变换里被冲淡。后面的层也很难判断：哪些信息应该保留，哪些信息应该修改。

所以 Transformer block 不是让子层直接替换原表示，而是让子层在原表示旁边写一份 "修改意见"。



### 残差连接：每层只写一份修改意见

这种 "原表示 + 修改意见" 的结构，就是 **残差连接（residual connection）**。

在现代 GPT 里，它通常不是直接把 `x` 丢进 Attention，而是先把 `x` 做一次归一化，再交给 Attention 读取：

```
x₁ = x₀ + Attention(LayerNorm(x₀))
x₂ = x₁ + MLP(LayerNorm(x₁))
```

 LayerNorm ：**进入 Attention 或 MLP 之前，先把当前向量整理到比较稳定的尺度。**

真正做内容更新的是后面的 Attention 和 MLP。它们的输出都不是 "新版本全文"，更像是 "我建议在原文上加这些改动"。

回到之前的句子：

```
小明 把 苹果 放进 书包 因为 它 很 重
```

处理 "它" 的时候，原始向量 `x₀` 里保留着 "它" 作为代词的基本信息。Attention 会拿这个位置去看前面的上下文，可能从 "小明"、"苹果"、"书包" 这些 token 里取信息。

Attention 输出的不是一个完整的新 token，而是当前 token 融合上下文之后得到的新信息。比如它可能会形成一份增量："它可能指代苹果"。

所以残差连接做的事情就是把这份增量加回原来的表示：

$$
x_1 = x_0 + \text{Attention}(\text{LayerNorm}(x_0))
$$

直觉上就是：

```
它
+ 看完上下文后，它可能指代苹果
= 更新后的它
```

也可以更口语化地写成：

```text
x₁ = 它 + 它可能是苹果
```

当然，真实模型里不是中文句子相加，而是两个 `d_model` 维向量相加。这个写法只是帮助我们抓住残差连接的直觉：**原来的表示还在，Attention 只是往上补了一份上下文判断。**

MLP 也是同样的思路。它拿到 `x₁` 之后，先经过 LayerNorm，再在当前位置内部继续加工信息，最后也不是彻底替换，而是继续往当前表示上加一份增量：

```text
更新后的它
+ 自己想完以后，进一步确认这个指代关系
= 再更新后的它
```

残差连接带来两个直接好处：

- **信息不容易被洗掉**：如果某一层暂时没学好，它只要输出接近 0，上一层的信息就能原样往后传。
- **多层更容易接起来**：因为每一层都是在原表示上加修改意见，而不是把前面的表示整份扔掉重写。



```
x → x + Attention(LayerNorm(x)) → x + MLP(LayerNorm(x)) → 下一个 block
```

这个结构让信息更容易一路传下去。哪怕某一层暂时没学出有用的修改，它也可以输出接近 0 的增量，让原来的表示继续往后走。



### LayerNorm：每次加工前先把数值拉稳

为什么每次进入 Attention 或 MLP 之前，都要先做 LayerNorm 这一步？

原因是：Attention 会混合不同 token 的信息，MLP 会重新组合同一个 token 内部的维度，残差连接又会把这些加工结果加回原向量。堆很多层之后，向量里的数值尺度很容易漂移。

有的层可能把某些维度越推越大，有的维度可能越来越弱。下一层拿到这样的输入，就像每次进来的音量都不一样：有时声音太大，有时声音太小，后面的处理就很难保持稳定。

所以在把向量送进 Attention 或 MLP 之前，通常会先做一次归一化。最经典的是 **LayerNorm（层归一化）**。

LayerNorm 对每个 token 的向量单独操作。假设某个 token 的表示是一个 `d_model` 维向量：

```
x = [x_1, x_2, ..., x_d]
```

LayerNorm 会先计算这个向量内部的均值和标准差：

```
μ = x 的均值
σ = x 的标准差
```

然后把它标准化：

```
x_normed = (x - μ) / σ
```

最后再用两个可学习参数做微调：

```
output = γ · x_normed + β
```

可以把它理解成：每次进加工车间前，先把向量的尺度拉回一个比较稳定的范围；至于哪些维度最后应该放大、哪些应该缩小，再交给可学习的 γ 和 β 去决定。

#### Pre-Norm vs Post-Norm

LayerNorm 放在哪里，也有讲究。

原版 Transformer 更接近下面这种 **Post-Norm**：

```
x → x + Sublayer(x) → LayerNorm
```

也就是先做子层，再把结果归一化。

现代 GPT / LLM 更常用的是 **Pre-Norm**：

```
x → x + Sublayer(LayerNorm(x))
```

也就是先归一化，再送进 Attention 或 MLP，最后仍然加回原来的 x。

Pre-Norm 的好处是：残差路径上始终保留一条比较直接的 `x + ...` 通道，梯度更容易一路传回前面的层。模型越深，这点越重要。

所以现代 GPT 的一个 block 通常写成：

```
x' = x  + Attention(LayerNorm(x))
y  = x' + MLP(LayerNorm(x'))
```

#### RMSNorm：更简单的归一化

LayerNorm 还有一个常见变体叫 **RMSNorm（Root Mean Square Norm）**。

它进一步简化了 LayerNorm：不再减均值，只用均方根 RMS 控制向量尺度：

```
x_normed = x / RMS(x)
output = γ · x_normed
```

它少做了一些计算，效果通常依然很好。LLaMA、GPT-NeoX、以及很多现代 LLM（包括这个项目）用的都是 RMSNorm。

不管是 LayerNorm 还是 RMSNorm，它们在这里承担的角色都很明确：**先把每个 token 的向量尺度拉稳，再交给 Attention 或 MLP 加工。**



### 完整的 Transformer Block

现在再回头看一个完整的 Transformer block，就不会只是 "Attention 接 MLP" 这么粗了。它里面有两条反复出现的小流程：

```text
先归一化 → 子层加工 → 加回原表示
```

把前面所有零件拼起来，一个 block 长这样：

```
                     ┌──────────────────────────┐
   x  ──────────────►│   Norm（LN/RMSNorm）     │
   │                 └──────────┬───────────────┘
   │                            ▼
   │                 ┌──────────────────────────┐
   │                 │  Multi-Head Attention    │
   │                 └──────────┬───────────────┘
   │                            ▼
   └──────────► (+) ◄───────────┘
                │
                ▼  (这里得到的中间结果记作 x')
   x' ──────────┐
   │            ▼
   │            ┌──────────────────────────┐
   │            │   Norm（LN/RMSNorm）     │
   │            └──────────┬───────────────┘
   │                       ▼
   │            ┌──────────────────────────┐
   │            │           MLP            │
   │            └──────────┬───────────────┘
   └──► (+) ◄──────────────┘
        │
        ▼
       下一个 block 的输入
```

用公式紧凑地写：

```
x' = x  + Attention(Norm(x))
y  = x' + MLP(Norm(x'))
```

一个 block 内部包含的可训练参数：

- Attention 部分：W_Q、W_K、W_V、W_O（输出投影）
- MLP 部分：W_1、W_2
- 两个归一化层的缩放参数（如果是 LayerNorm，还会有偏移参数 β；如果是 RMSNorm，通常只有缩放参数 γ）

这就是 GPT 的 **基本积木**。Attention 负责跨 token 交换信息，MLP 负责每个 token 自己加工信息，残差连接负责把修改意见接回原表示，归一化负责让每次加工前的数值尺度更稳定。

整个模型，就是把这种积木堆 N 次。



### 堆 N 层：为什么深度有意义

一个 block 已经很强了：每个 token 看了一轮上下文，又独立思考了一轮。但只用一层是不够的。

把 block 堆成 N 层之后，会发生一件有意思的事：**不同层学到的东西不一样**。

研究者通过可视化 / 探针实验大致观察到：

- **底层（接近输入）**：更倾向于学习 **局部、表层** 的模式 —— 拼写、词形、相邻搭配。
- **中层**：开始学 **句法结构** —— 谁是主语、谁是宾语、从句边界在哪。
- **高层（接近输出）**：开始学 **语义、指代、逻辑关系** —— "它" 指谁、上下文意图是什么、要预测的下一个 token 大概是什么类型。

这并不是人手工分配的，而是数据驱动下自然涌现的分工。每一层在前一层的 "理解" 基础上，再加一层 "再理解"，层层累积起来才形成模型对一段文本的完整把握。

GPT 不同规模就是在这里拉开差距的：

```
小模型：n_layers = 6  / 12
中模型：n_layers = 24 / 32
大模型：n_layers = 48 / 80 / 96 ...
```

层数、d_model、num_heads 这几个人工设定的超参数一起决定了模型的容量和算力开销。

### 模型怎么知道文本 "顺序"？

前面已经讲过：Tokenizer 把文本切成 token id，Embedding 表把每个 id 查成一个 d_model 维向量。现在输出端也已经接上了：多层 block 的最终表示会经过 lm_head 变成 logits，再经过 softmax 变成下一个 token 的概率。

但第一个 block 的输入还有一个问题悬着：**只有 token Embedding，真的够吗？**

但回顾一下 Attention 公式：
$$
\text{Attention}(Q, K, V) = \text{softmax}\left(\frac{QK^T}{\sqrt{d_k}}\right) V
$$

这个公式里有任何 "位置" 的概念吗？

完全没有。Q、K、V 都是按矩阵的行存的，QK^T 算的是任意两行之间的相似度，**它根本不在乎哪一行排在哪一行前面**。

做个思想实验：把句子 token 顺序打乱再喂进去，Attention 算出的每个位置的输出，跟原来一模一样，只是跟着位置一起被搬了位置。

也就是说：

**Attention 本身是顺序无关的。它把输入当成 "一袋 token"。**

但语言显然讲顺序。"小明打了张三" 和 "张三打了小明" 是两回事。所以我们必须主动把位置信息塞进 token 向量里，否则 Attention 永远学不会 "谁先谁后"。

#### 朴素做法：可学习的位置 Embedding

最直接的做法（GPT-2 用的）：开一张和 token Embedding 平行的 **位置 Embedding 表**：

```
position_Embedding[0]  →  位置 0 的向量
position_Embedding[1]  →  位置 1 的向量
...
position_Embedding[n]  →  位置 n 的向量
```

然后把它**加到** token Embedding 上：

```
x_i = token_Embedding(token_i) + position_Embedding(i)
```

同一个 token 出现在不同位置，向量就不一样了，Attention 就能区分。

这个做法很简单，但有个限制：位置 Embedding 表只能覆盖训练时见过的最大长度。超过这个长度，模型就不知道怎么编码位置了。

#### 现代做法：RoPE（旋转位置编码）

LLaMA / GPT-NeoX 用的都是 **RoPE（Rotary Positional Embedding）**。

它的核心思路很巧：不 "加" 位置向量，而是 **直接对 Q、K 做旋转**，旋转角度由位置决定。具体地，对每对相邻的两个维度，看作复平面上的一个点，按位置 i 把它逆时针转 i × θ 角度。

为什么这么设计是好的：

- 位置信息只进 Q、K（影响相似度计算），**不进 V**（不污染要传递的内容）。
- 两个 token 的相对位置差 j − i 会自动反映在 Q_i · K_j 的内积里，意味着 Attention 天生就能感知 **相对位置**。
- 比绝对位置 Embedding 更容易外推到训练时没见过的长序列。

RoPE 的具体推导和实现细节暂时不展开，这里只要记住：

**位置信息是通过对 Q、K 做 "按位置旋转" 注入到 Attention 里去的。**



## 预训练：让模型先学会续写世界

### 怎么把整段文本 "喂" 给模型

现在我们已经知道了单个位置如何从 token 向量一路走到 logits、softmax 和 loss，也知道 loss 会反过来更新参数。

但真正训练时，还有一个效率问题：**一整段文本要怎么喂给模型？**

### 一次喂整句话，让每个位置都做一次预测

最朴素的训练方式听上去应该是：

| 看到                     | 预测        |
| ------------------------ | ----------- |
| 看到 "小明"              | 预测 "把"   |
| 看到 "小明 把"           | 预测 "苹果" |
| 看到 "小明 把 苹果"      | 预测 "放进" |
| 看到 "小明 把 苹果 放进" | 预测 "书包" |

每条样本跑一次前向 + 一次反传，4 个 token 跑 4 次。但这样太低效了 —— 一段长度为 n 的文本要跑 n 次前向。

大聪明：**一次性把整段文本喂进去，让模型在每个位置都同时预测下一个 token。** 一次前向，拿到所有位置的预测，反传一次梯度。

但这样会出大问题。

### Causal Mask：训练时不能偷看未来

在算 Attention 的时候，每个 token 默认都能看到序列里 **所有** 其他 token，包括它后面的。

也就是说，当我们让位置 2 的 "把" 去预测位置 3 的 "苹果" 时，"把" 这个位置的 Attention 会同时看到 "苹果"、"放进"、"书包" …… **答案就明明白白写在输入里**，模型完全可以作弊：直接从下一个位置抄答案，loss 就是 0。这种训练学不到任何语言规律。

要让 "一次喂整句" 和 "预测下一个 token" 这两件事兼容，必须强制规定：

> 位置 i 的 token，只允许 attend 到位置 ≤ i 的 token。

具体做法很简单 —— 在算完 QK^T 之后、过 softmax 之前，把所有 "未来位置" 的注意力分数设成 −∞：

|          | 位置1 | 位置2 | 位置3 | 位置4 | 位置5 |
| -------- | ----- | ----- | ----- | ----- | ----- |
| 位置1的Q | ✓     | −∞    |       |       |       |
| 位置2的Q | ✓     | ✓     | −∞    |       |       |
| 位置3的Q | ✓     | ✓     | ✓     | −∞    |       |
| 位置4的Q | ✓     | ✓     | ✓     | ✓     | −∞    |
| 位置5的Q | ✓     | ✓     | ✓     | ✓     | ✓     |

−∞ 经过 softmax 之后变成 0，所以未来位置在加权平均时贡献为 0，等价于 "看不见"。

这就是 **因果掩码（Causal Mask）**，一个下三角矩阵。每个位置只能看自己和前面的，看不到后面的。

加上 causal mask 之后：

- **训练时一次前向就能完成所有位置的 "预测下一个 token" 任务**，且不会作弊。
- **每个位置的 loss 是独立的**，整段 loss 就是所有位置 loss 的平均，反传一次梯度更新所有参数。

这就是 GPT 这种 **decoder-only** 架构的核心训练设定。

> 顺便说一句：BERT 那种 encoder 模型 **不用** causal mask，因为它的训练目标不是 "预测下一个"，而是 "完形填空"（随机遮住一些词让模型猜），允许双向看上下文。是任务决定了要不要 mask，而不是相反。

到这里，我们已经知道 GPT 的单次训练目标了：

```text
给定前文 → 预测下一个 token
```

但真正的预训练不是拿一句话练几次，而是把海量文本切成 token，再一批一批喂给模型。模型像一个永远在做填空题的学生：

```text
今天 天气 很 ___
Transformer 的 核心 是 ___
def add(a, b): return ___
```

每做错一次，就根据答案改一点参数。做了几百亿、几千亿次之后，它慢慢形成一种压缩在参数里的语言直觉：什么词常跟什么词出现，代码结构大概怎么写，数学表达式长什么样，文章的逻辑怎么展开。

这一步得到的模型，叫 **base model**。

base model 很重要，但它还不是聊天助手。它更像一个读了很多书、会模仿文本继续往下写的人。我们输入：

```text
请解释一下牛顿第二定律
```

它可能会继续写一个回答，也可能继续写下一轮对话，甚至可能补出一段奇怪的网页文本。因为在预训练阶段，它学到的是 "文本接下来通常长什么样"，而不是 "作为助手应该怎样回答用户"。

### 预训练数据：模型到底在读什么

模型吃进去的 "海量文本"，是很多来源混在一起的语料：网页、百科、书籍、论文、代码、数学题、论坛、问答、文档、多语种文本等等。

```text
模型能力的上限，很大一部分来自它预训练时见过什么。
```

如果数据里全是低质量网页、广告页、重复模板、SEO 垃圾，模型就会很会模仿这些东西。如果数据里有高质量教材、技术文档、代码、数学推导、科学论文、严肃问答，它才更可能学到稳定的知识结构和推理格式。

所以现代 LLM 的预训练数据，通常不是 "把互联网全倒进去"，而是一套反复清洗、去重、打分、混合的工程。

### FineWeb

**FineWeb** 是 Hugging Face 从 Common Crawl 这类大规模网页抓取中清洗出来的高质量网页文本集合。

Common Crawl 本身像一座原始矿山：里面有新闻、博客、论坛、文档，也有广告、导航栏、乱码、重复页、机器生成页。FineWeb 做的事情，就是从这座矿山里尽量筛出更像正常文章、更适合训练语言模型的网页文本。

**FineWeb-Edu** 则是在 FineWeb 的基础上，再偏向筛选 "教育价值更高" 的文本。比如解释性文章、教程、百科式内容、学习材料，会比普通网页更受青睐。常用的 `FinewebEdu-100B` 就是：

```text
FineWeb-Edu：偏教育内容的网页语料
100B：大约 1000 亿 token 规模
```

它的优点是干净、解释性强、适合小模型快速学到通用语言和基础知识。缺点也很明显：如果只吃教育网页，代码、数学、科学、论坛对话、多语种等能力会受限。

### ClimbMix

**ClimbMix** 是 NVIDIA Nemotron 系列使用思路里的一个预训练数据混合。它不只是网页，而是把多种高价值来源混在一起，例如高质量 web text、代码、数学、科学和其他知识密集型文本。

`ClimbMix-400B` 是我们要使用的预训练数据集：

```text
ClimbMix：一个更综合的预训练数据配方
400B：大约 4000 亿 token 规模
```

### 常见的英文网页和通用语料

现在很多开源模型的预训练数据，都会从 Common Crawl 派生数据开始。常见名字包括：

- **C4**：Google 从 Common Crawl 清洗出的英文网页语料，早期 T5 训练常用。
- **The Pile**：EleutherAI 做的混合语料，包含网页、书籍、论文、代码、问答等多种来源。
- **RedPajama**：复现 LLaMA 风格数据配方的开源混合数据。
- **RefinedWeb**：Falcon 系列背后的大规模网页清洗语料。
- **Dolma**：AllenAI / OLMo 使用的开放预训练语料。
- **DCLM**：DataComp for Language Models，用系统化实验寻找更好的数据过滤和混合方法。
- **FineWeb / FineWeb-Edu**：Hugging Face 推出的高质量网页语料和教育子集。

这些数据集名字背后，其实都是同一个大问题：

```text
从混乱的互联网文本里，怎样筛出更值得模型学习的部分？
```

### 中文常用预训练数据

中文模型也需要中文语料，只靠英文数据是无法 "顺便学会" 中文的。中文的分词、标点、表达习惯、知识来源、网页结构都不一样。

常见的中文或含中文数据来源包括：

- **中文网页语料**：从 Common Crawl、中文站点、新闻、论坛、博客中清洗出的中文文本。
- **百科类语料**：中文 Wikipedia、百科条目、知识库文本，适合提供结构化常识。
- **书籍和长文档**：小说、教材、说明文、报告，适合学习长文本结构，但版权和授权要非常小心。
- **问答和社区语料**：知乎类问答、论坛、技术社区，能提供口语化解释和问题格式，但质量差异很大。
- **开源中文数据集**：例如 WuDao、CLUECorpus、MNBVC、SkyPile、WanJuan 等，都可以看作中文或中英混合预训练语料探索的一部分。

很多中文网页里会混入广告、采集站、乱码、繁简混杂、机器翻译、重复转载。对小模型来说，这些噪声影响很大，因为它容量有限，吃进去的每一点坏数据都会占位置。

### 代码数据

如果希望模型会写代码，预训练里就必须有足够多的代码和技术文档。

常见代码数据来源包括：

- **GitHub 开源仓库**：最主要的代码来源，但要处理许可证、重复文件、生成文件、压缩产物、vendored 依赖等问题。
- **The Stack / StarCoderData**：BigCode 社区整理的大规模开源代码数据。
- **CodeSearchNet**：带有函数、文档字符串和自然语言描述的代码数据。
- **技术文档和 API 文档**：让模型学会库的用法、参数含义和示例格式。
- **编程问答和教程**：让模型学会解释错误、补全代码、比较方案。

代码数据和普通文本不太一样。它对格式、缩进、符号、长距离依赖更敏感。一个括号错了，整段程序可能就不能运行。所以代码能力不只是 "多背 API"，还需要模型在预训练中反复看到真实项目结构、函数调用、测试、错误处理和文档说明。

### 数学、科学和论文数据

如果希望模型在科学和数学上更可靠，只靠普通网页也不够。

常见来源包括：

- **arXiv 论文**：数学、物理、计算机科学等领域的论文文本，公式和专业表达很多。
- **PubMed / PMC**：生物医学论文和摘要，适合医学和生命科学语言。
- **OpenWebMath**：从网页中抽取的数学相关文本。
- **Proof-Pile / Proof-Pile-2**：偏数学证明、形式化推理、论文和教材的混合数据。
- **教材、讲义、题解、竞赛题**：有助于模型学习分步骤推理和严谨表述。

这些数据能提供更强的专业表达，但也有挑战。论文里公式、LaTeX、引用、表格、脚注很多，清洗不好会变成噪声；专业文本也更容易带来 "看起来很懂" 的幻觉，因为模型学到了术语和语气，不等于真的会验证结论。

### 多语种数据

多语种模型还会混合不同语言的数据。常见来源包括：

- **mC4**：多语种 Common Crawl 清洗语料。
- **OSCAR**：从 Common Crawl 里按语言分类整理的多语种语料。
- **CC100**：覆盖很多语言的大规模网页语料。
- **CulturaX**：较新的多语种大规模清洗语料。
- **ROOTS**：BigScience / BLOOM 背后的多语种数据工程。

多语种训练有一个很现实的问题：配比怎么定。

如果英文比例太高，模型会更强但小语种能力容易被挤压；如果每种语言平均分配，高资源语言的数据质量优势又用不满。很多训练会使用温度采样这类方法：让低资源语言被适当放大，但又不至于让少量重复数据被模型背烂。

### 数据配方比单个数据集更重要

真正训练 GPT 时，通常不会只选一个数据集，而是做一个 mixture：

```text
高质量网页 + 教育文本 + 百科 + 代码 + 数学 + 科学论文 + 多语种 + 少量社区问答
```

每一类数据都在塑造不同能力：

- **网页和百科：**提供世界知识和普通语言表达；
- **教育文本：**提供解释能力；
- **代码：**提供结构化生成和符号操作能力；
- **数学和科学：**提供严谨推理、公式、专业概念；
- **多语种：**提供跨语言表达和翻译能力；
- **问答和论坛：**提供真实用户问题的形状。

但比例不是越平均越好。一个模型如果代码吃太多，可能普通聊天变得像技术文档；如果论坛吃太多，可能口语化强但严谨性下降；如果论文吃太多，可能措辞专业但不够亲切。数据配方就像给模型安排课程表：什么课上多久，先上什么、后上什么，都会影响它最后长成什么样。

### 数据不是一篇篇喂进去，而是打包成训练批次

原始训练文本通常来自网页、书籍、代码、百科、论坛等不同来源。它们先被 Tokenizer 变成 token id，然后再被打包成固定长度的小段。

可以把预训练数据想成一条非常长的传送带：

```text
文档 A 的 token → 文档 B 的 token → 文档 C 的 token → ...
```

模型训练时不是每次只拿一个 token，而是每次切一块下来，比如 2048 个 token：

```text
[token_0, token_1, token_2, ..., token_2047]
```

然后训练目标就是整体右移一格：

```text
输入：token_0, token_1, token_2, ..., token_2046
答案：token_1, token_2, token_3, ..., token_2047
```

这就像老师把一整页练习题发给学生。学生不是只回答最后一道题，而是每个位置都要回答：

```text
看到第 0 个 token，猜第 1 个
看到第 0~1 个 token，猜第 2 个
看到第 0~2 个 token，猜第 3 个
...
```

一页纸上有很多小题，模型一次前向就能把这些小题都做完。这样训练效率才够高。

### BOS：告诉模型这里是新文档的开始

GPT 的训练数据里，每个新文档前面会放一个特殊 token：

```text
<|bos|>
```

BOS 是 Beginning of Sequence，也就是 "序列开始"。

它像一张新试卷左上角的标题：告诉模型，前面的上下文到这里断开了，这里是新的文本起点。否则如果我们把很多文档直接拼在一起，模型可能会误以为：上一篇新闻的结尾和下一篇代码教程的开头本来就应该连在一起。

没有 BOS 的时候，训练数据像一本被撕掉章节标题又强行粘在一起的书。模型当然也能读，但会多学到一些奇怪的跨文档连接。

有了 BOS，模型至少知道：

```text
这里重新开始了。
```

这对预训练很重要，也对后面的聊天格式很重要。

### Packing：尽量别浪费上下文窗口

模型一次最多能看多少 token，叫 **上下文长度**。比如 `max_seq_len = 2048`，意思是一次训练最多处理 2048 个连续 token。

但真实文档长短不一：

```text
文档 A：120 token
文档 B：800 token
文档 C：3000 token
文档 D：50 token
```

如果每篇文档都单独占一个 2048 长度的训练样本，短文档会浪费大量空位。就像搬家时每个纸箱都只放一件小东西，车厢很快就被 "空气" 填满了。

所以训练系统会做 packing：把多个文档尽量塞进同一个上下文窗口里。

直觉上像装行李箱：

```text
2048 token 的箱子
├── 文档 A：120 token
├── 文档 B：800 token
├── 文档 D：50 token
└── 再塞一些能放得下的文档
```

这里用的是一种 best-fit 思路：从缓冲区里挑尽量合适的文档塞进去，让每一行训练样本尽可能装满。这样 GPU 做的计算更少浪费，训练速度也更高。

### Batch：一次不是训练一句，而是训练一摞

如果单条训练样本是一页练习题，那么 batch 就是一摞练习题。

```text
batch = 32
seq_len = 2048
```

可以理解成一次给模型发 32 页试卷，每页 2048 个 token。模型并行把这些题都做完，然后把所有错误汇总成一个平均 loss，再统一更新一次参数。

batch 太小，训练会像只看几道题就改一次学习方向，容易抖。batch 大一点，模型能看到更多样本后再更新，方向通常更稳。

但 batch 越大，需要的显存越大。所以实际训练里常用 **梯度累积（gradient accumulation）**：

```text
先做第 1 小批，记下梯度
再做第 2 小批，继续累积梯度
再做第 3 小批，继续累积梯度
攒够以后，再统一更新参数
```

这就像老师不是看完一张卷子就立刻改教学计划，而是先看完一叠卷子，统计大家最常错在哪里，再调整下一节课怎么讲。

### 多卡训练：很多学生同时批改不同卷子

如果有多张 GPU，就可以让每张 GPU 负责一部分 batch。

比如 8 张 GPU：

```text
GPU 0 看第 0 批数据
GPU 1 看第 1 批数据
GPU 2 看第 2 批数据
...
GPU 7 看第 7 批数据
```

每张卡都算出自己那部分数据带来的梯度。然后大家把梯度同步一下，得到一个统一的平均方向，再一起更新同一份模型参数。

这很像 8 个助教同时批改不同班级的卷子，最后开会汇总：

```text
哪些题错得最多？
哪些知识点最薄弱？
下一轮教学该往哪里调整？
```

模型训练中的分布式数据并行，本质上就是这种分工协作。

### 优化器：不是知道错了就知道怎么改

loss 告诉模型：

```text
你错了多少。
```

反向传播告诉模型：

```text
每个参数大概要往哪个方向改。
```

但真正决定 "改多大、怎么改、要不要带一点惯性" 的，是优化器。

可以把训练想成在雾里下山。loss 是海拔，目标是走到更低的地方。梯度告诉你脚下哪边更陡，但它不保证你一步迈多大最合适。

- 步子太小：下山很慢；
- 步子太大：可能冲过头，甚至摔死；
- 只看当前坡度：可能在崎岖地形里来回震荡；
- 带一点历史惯性：往往走得更稳。

AdamW、Muon 这类优化器，就是在回答：

```text
既然知道每个参数的梯度了，下一步到底该怎么走？
```

GPT 框架里不是所有参数都用同一种走法。Embedding、lm_head、一些标量参数更像词表记忆和控制旋钮；Transformer 里的大矩阵更像主要的信息变换规则。不同类型的参数数值行为不一样，所以会把它们分组，用不同的学习率和优化策略更新。

这不影响我们理解主线。主线仍然是：

```text
做题 → 算错多少 → 反推怎么改 → 优化器更新参数 → 下一轮继续做题
```

预训练就是把这个循环重复很多很多次。

### 学习率：每一步迈多大

优化器里最重要的旋钮之一是 **学习率（learning rate）**。

学习率可以理解成每次改参数的步幅。

训练刚开始时，模型参数是随机的。如果一上来步子太大，很容易把模型推到不稳定的位置。所以很多训练会先 **warmup**：

```text
刚开始小步走
确认方向稳定后
再逐渐加到正常步幅
```

训练快结束时，又常常会 **warmdown**：

```text
前面大步找方向
后面小步精修
```

这有点像雕刻。刚开始用大刀削出形状，中间用正常工具修结构，最后用细砂纸慢慢磨。预训练不是一步到位，而是一段节奏被精心安排的长跑。



## 评估：怎么知道模型真的学到了东西

训练时 loss 在下降，看起来很美好。但问题是：

**loss 下降，真的等于模型变聪明了吗？**

不一定。

一个学生在练习册上分数提高，可能是真的理解了，也可能只是熟悉了题型。语言模型也一样。我们需要从不同角度评估它。

### BPB (Bits per Byte)：模型压缩文本的能力

前面讲 Tokenizer 时已经提到过 BPB (Bits per Byte)。

它可以理解成：

```text
模型平均需要花多少信息量，才能描述原始文本里的每个字节。
```

如果 BPB 越低，说明模型对文本越有把握。它越能提前猜到接下来会出现什么，描述这段文本需要付出的惊讶程度就越低。

可以用一个压缩软件类比。

如果一段文本很有规律：

```text
哈哈哈哈哈哈哈哈哈哈哈哈
```

压缩软件很容易压得很小，因为它知道后面大概率还是 "哈"。

如果一段文本完全随机：

```text
Q7x#pL9!zA0@vM2
```

就很难压缩，因为每个字符都很意外。

语言模型也是类似。如果它真的理解了语言模式，那么真实文本对它来说就不那么意外，BPB 就会下降。

BPB 的好处是，它比 "每 token loss" 更公平。因为不同 Tokenizer 的 token 大小不一样，有的 token 像大箱子，有的 token 像小盒子。按 byte 统一计量，就像物流按公斤收费，而不是按箱子个数收费。

### Sample：让模型讲两句

BPB 很重要，但它毕竟是一个数字。数字能告诉我们模型是否更会预测文本，却不一定能告诉我们它 "说出来" 是什么味道。

所以训练过程中常常会做采样：给模型一个开头，让它续写。

这像考试以外的课堂提问。卷面分数是一回事，站起来讲两句又是另一回事。

一个模型可能 BPB 还不错，但生成时出现：

- 重复同一句话；
- 话题突然漂移；
- 标点和格式很乱；
- 看似流畅但内容空洞。

这些问题，人类扫一眼样例往往比看曲线更容易发现。

采样不是严格指标，是训练时很重要的体感检查。

### CORE：不只是续写，还要会做任务

CORE 是 DCLM 里使用的一组综合评估指标，DCLM 的全称是 **DataComp for Language Models**，意思大概是 "给语言模型做数据竞赛"。它关注的问题不是再发明一个新模型结构，而是系统比较：同样的模型、同样的训练预算下，什么数据过滤方法、什么数据混合配方，能训练出更强的语言模型。

可以把它理解成一组任务考试的总分。它不是让模型继续预测普通文本，而是拿阅读理解、常识、分类、问答、符号推理等任务来测 base model 的泛化能力。

base model 的核心训练目标是续写，但我们更关心的是：

```text
它能不能把读到的模式迁移到具体任务上？
```

比如给它几个示例：

```text
问题：法国首都是哪里？
答案：巴黎

问题：日本首都是哪里？
答案：东京

问题：德国首都是哪里？
答案：
```

如果模型能接出 "柏林"，说明它不只是机械续写，还能从上下文里看懂任务格式，并调动预训练中学到的知识。

CORE 这类评估就像综合考试：不是只看学生会不会背下一句话，而是看它能不能在阅读理解、常识、分类、问答等任务上表现出泛化能力。

在 GPT 的训练路线里，BPB 更像 "语言建模内功"，CORE 更像 "综合做题能力"，sample 更像 "现场口头表现"。

三者合起来，才比较像一个完整的体检：

```text
BPB：基本功有没有变强
CORE：任务能力有没有变强
Sample：生成体感有没有变好
```

### Base model 的边界

有了数据、模型架构，就可以训练出 base model 了。

```
- depth: 34
- max_seq_len: 2048
- target_param_data_ratio: 40
- device_batch_size: 4
- total_batch_size: 524,288
- Number of parameters: 2,217,082,880 (2.2B)
- Number of FLOPs per token: 1.426509e+10
- Calculated number of iterations: 169,150
- Number of training tokens: 88,683,315,200
- Tokens : Params ratio: 40.0000
- DDP world size: 4
- Minimum validation bpb: 0.7045
- Final validation bpb: 0.7045
- MFU %: 47.40%
- Total training flops: 1.265075e+21
- Peak memory usage: 69811.85MiB
```

`depth = 34` 表示模型堆了 34 层 Transformer block。每一层都包含一组 Attention 和一组 MLP。层数越多，模型能做的多轮信息加工越多，但参数量和计算量也会一起上升。

`max_seq_len = 2048` 表示每条训练样本最长 2048 个 token。训练时模型一次看到的是一段长度为 2048 的 token 序列，并在每个位置预测下一个 token。它也决定了 causal attention 最大能看多长的上下文。

`target_param_data_ratio = 40` 表示目标训练 token 数大约是参数量的 40 倍。这个数字来自 scaling law 的直觉：模型大小和训练数据量要匹配。模型太大但 token 太少，会学不满；模型太小但数据太多，后面收益会变低。这里我们选择的是：

```text
目标训练 token 数 = 参数量 × 40
```

`device_batch_size = 4` 表示每张 GPU 每次前向/反向处理 4 条样本。注意这里的单位是“样本条数”，每条样本长度是 2048 token，所以单张卡一次 micro-batch 处理的 token 数是：

```text
4 × 2048 = 8192 tokens
```

`DDP world size = 4` 表示用了 4 张 GPU 做分布式数据并行。所以所有 GPU 合起来，一次 micro-step 处理：

```text
8192 × 4 = 32,768 tokens
```

`total_batch_size = 524,288` 表示一次真正的 `optimizer.step()` 前，总共累计 524,288 个 token 的梯度。因为一次 micro-step 只有 32,768 token，所以需要做梯度累积：

```text
grad_accum_steps = 524,288 / 32,768 = 16
```

也就是说，每张卡连续做 16 次前向/反向，把梯度攒起来，然后 4 张卡同步，最后才更新一次参数。

`Calculated number of iterations = 169,150` 表示总共更新参数 169,150 次。这个数来自目标 token 数除以 total batch size：

```text
Number of training tokens = 2,217,082,880 × 40
           = 88,683,315,200

iterations = 88,683,315,200 / 524,288
      = 169,150
```

所以 `Number of training tokens = 88,683,315,200`，也就是大约 886 亿 token。`Tokens : Params ratio = 40.0000` 正是因为：

```text
88,683,315,200 / 2,217,082,880 = 40
```

`Number of FLOPs per token = 1.426509e+10` 是估算训练时每处理一个 token 大约需要多少浮点运算。它包括前向、反向、矩阵乘法和 attention 相关计算。总训练计算量就是：

```text
Total training flops = FLOPs per token × training tokens
           ≈ 1.426509e10 × 88,683,315,200
           ≈ 1.265075e21
```

`MFU = 47.40%` 是 Model FLOPs Utilization，表示实际训练吞吐达到了硬件理论峰值的大约 47.4%。它不是模型质量指标，而是训练系统效率指标。

`Peak memory usage = 69811.85MiB` 表示单卡峰值显存大约 68.2 GiB。这里面不只有模型参数，还包括梯度、优化器状态、激活值、KV/attention 中间张量、通信 buffer 等。

`Minimum validation bpb = 0.7045` 和 `Final validation bpb = 0.7045` 是验证集上的 BPB 指标。BPB 越低，说明模型越能压缩/预测验证文本。这里最小值和最终值一样，表示训练结束时刚好达到这次训练过程中的最好验证 BPB。

### 参数量是怎么算出来的

模型的参数量 `Number of parameters = 2,217,082,880 (2.2B)` 是怎么来的。

这次模型的 tokenizer 词表大小是：

```text
vocab_size = 32768
```

nanoChat 会先根据 `depth` 推导隐藏维度。默认宽深比是：

```text
aspect_ratio = 64
head_dim     = 128
```

所以：

```text
d_model = depth × aspect_ratio
   = 34 × 64
   = 2176
```

2176 刚好可以被 128 整除，因此注意力头数是：

```text
n_head = d_model / head_dim
       = 2176 / 128
       = 17
```

也就是说，这个模型有 34 层，每层 hidden size 是 2176，每层有 17 个 attention head，每个 head 维度是 128。

先看一个 Transformer block 的参数量。

Attention 里有 4 个主要矩阵：

```text
Wq:     d_model × d_model
Wk:     d_model × d_model
Wv:     d_model × d_model
Wo:     d_model × d_model
```

这里没有 bias，所以 Attention 参数量是：

```text
Attention params per layer
= 4 × d_model × d_model
= 4 × 2176 × 2176
= 18,939,904
```

MLP 采用的是常见的 `d_model → 4d_model → d_model` 结构，所以有两个矩阵：

```text
W1: d_model  × 4d_model
W2: 4d_model × d_model
```

参数量是：

```text
MLP params per layer
= d_model × 4d_model + 4d_model × d_model
= 8 × d_model × d_model
= 8 × 2176 × 2176
= 37,879,808
```

所以一个 Transformer block 的主干矩阵参数量是：

```text
Block params
= Attention params + MLP params
= 18,939,904 + 37,879,808
= 56,819,712
```

34 层合起来：

```text
Transformer blocks params
= 56,819,712 × 34
= 1,931,870,208
```

然后看词表相关的矩阵。输入端 token embedding 是：

```text
token embedding params
= vocab_size × d_model
= 32768 × 2176
= 71,303,168
```

输出端 `lm_head` 也把 `d_model` 映射回词表大小：

```text
lm_head params
= d_model × vocab_size
= 2176 × 32768
= 71,303,168
```

这次统计口径里还包含两张同样大小的词表级辅助矩阵。它们可以理解成额外的 embedding / value embedding 类参数，每张大小同样是：

```text
32768 × 2176 = 71,303,168
```

因此，词表级矩阵一共是 4 张：

```text
vocab-related params
= 4 × 71,303,168
= 285,212,672
```

最后总参数量就是：

```text
Total params
= Transformer blocks params + vocab-related params
= 1,931,870,208 + 285,212,672
= 2,217,082,880
```

也就是 22.17 亿参数。

我们设定了 `target_param_data_ratio = 40`，所以训练预算直接按参数量线性放大，训练 token 数会是 886 亿左右。

34 层 Transformer blocks 基础模型的 CORE 得分为 0.3382。对比而言，20 层的模型偏速成，GPT-2 大概是 30 层左右。

| 模型 | 参数量 | CORE 得分 |
| --- | ---: | ---: |
| d20 | 0.56B | 0.22 |
| GPT-2 | 1.6B | 0.25 |
| d32 | 1.88B | 0.3168 |
| d34 | 2.2B | 0.3382 |


我们可以把 base model 跑起来看一眼：

它的交互方式更像 "给一段开头，让模型接着写"：

```text
Prompt: Why sky is blue?

Completion: Why Sun is yellow and Moon is black? Why stars are small and planets are big? These are some of the questions which you have been asking yourself. To fiwe have to study light. Light is electromagnetic radiation of a certain wavelength. (I think that this is a new word for you; try to pronounce it). This means that lf by something which emits heat or energy. It is a form of radiation. The difference is that stars, in most cases, are much hotter than Sun. Moon is much cooler thancolors are different. The electromagnetic radiation is white. When it is...
```

这个输出很有代表性。我们问的是 "why sky is blue?"，但 base model 只是沿着当前文本分布往后续写：既可能接出一个解释，也可能接成一串相似问题，还可能接成网页、问答样例、文章片段。

这是 base model 的本职工作：**预测接下来最像训练语料的 token**。

**预训练模型已经很会续写，但它还不会聊天 (Chat)。**

它知道很多语言模式，也学到大量知识，但它没有被明确训练成一个 "听用户请求并回答" 的角色。

如果说预训练像通识教育，那么下一步 SFT 就像职业培训：把这个会读书、会续写的人，训练成一个会按对话格式工作的助手。

## SFT：把会续写的模型训练成聊天助手

SFT 的全称是 **Supervised Fine-Tuning**，监督微调。

SFT 阶段，模型看到的是一条条整理好的对话：

```text
用户：请解释一下什么是 Transformer
助手：Transformer 是一种基于注意力机制的神经网络架构……
```

这一步的目标不是让模型从零学语言，而是告诉它：

```text
当输入长成用户提问时，你应该像助手一样回答。
```

可以把 base model 想成一个读了很多书的人。SFT 像把他带到客服岗位上培训：

- 用户说话时，要听完问题；
- 轮到助手时，要给出回答；
- 不要把用户和助手的角色搞混；
- 不要续写一篇网页，而是完成这轮对话。

### 全参数微调 (Full Fine-Tuning)

全参数微调需要先加载预训练阶段得到的 base checkpoint：

```text
base model checkpoint
→ 继续用 SFT 数据训练同一个模型
→ 保存成 chatsft checkpoint
```

训练时，模型原来的 Embedding、Attention、MLP、lm_head 等参数都会继续参与更新。也就是说，base model 不是被固定住的；SFT 数据会直接改写整个模型的参数分布，让它从 "会续写文本" 变成 "会按用户请求回答"。

具体的过程是：加载 base model，构建原模型参数的 optimizer，然后用对话数据继续反向传播。脚本还可以从 base 训练的 optimizer checkpoint 里加载动量状态，再把学习率重置成 SFT 阶段的新学习率。

所以全参数的 Chat SFT 可以理解为：

```text
把整个 base model 接着训练一小段，让它整体适应聊天格式。
```

这种方法很直接，模型容量也完全打开；代价是训练时要保存全部参数的梯度和优化器状态，显存开销会比较大。

### LoRA：给大矩阵加一个小镜头

**LoRA（Low-Rank Adaptation）** 来自论文 *LoRA: Low-Rank Adaptation of Large Language Models*，作者包括 Edward J. Hu、Yelong Shen、Phillip Wallis、Zeyuan Allen-Zhu、Yuanzhi Li、Shean Wang、Lu Wang、Weizhu Chen。这篇论文的核心问题很朴素：

> 大模型已经很会说话了。微调一个新任务时，我们真的需要改动全部参数吗？

论文的观察是：微调带来的参数改变量，往往有很低的 **Intrinsic rank（内在秩）**。换句话说，模型学新任务时，并不是每个方向都需要大幅移动；真正有用的变化，可能集中在少数几个方向上。

还记得前面讲 Attention 时的 "摄像机" 类比吗？`W_Q`、`W_K`、`W_V` 这些线性层，就像把同一个 token 从不同角度拍成 Query、Key、Value 的摄像机。

全参数微调相当于：

```text
把整台摄像机重新调一遍
```

LoRA 的做法更像：

```text
原摄像机不动
只在镜头前加一个可训练的小转接环
让画面沿少数几个关键方向微调
```

这就是 "Low-Rank Adaptation" 这个名字的意思：低秩，不是低级，而是说这个改变量只允许沿着少数独立方向变化。

我们从一个普通线性层开始看：

```text
y = W x
```

这里的维度可以写成：

```text
x ∈ R^d
W ∈ R^(k × d)
y ∈ R^k
```

`d` 是输入向量的宽度，`k` 是输出向量的宽度。比如在 Attention 里，某个 token 的隐藏状态 `x` 先经过 `W_Q`，变成 Query；这里的 `d` 就是输入 hidden size，`k` 就是 Query 投影后的宽度。

如果 `W` 是 `4096 × 4096`，那么这一台 "摄像机" 就有：

```text
4096 × 4096 = 16,777,216
```

约 1677 万个参数。全参数微调等价于直接学习一个同样大的改变量：

```text
W' = W + ΔW
```

问题是，`W` 有多大，`ΔW` 就有多大。一个 Transformer 里有很多这样的矩阵：Attention 里的 `W_Q`、`W_K`、`W_V`、`W_O`，MLP 里的上投影和下投影。全部训练时，梯度、optimizer state、checkpoint 都会跟着变大。

LoRA 不直接训练完整的 `ΔW`，而是把它拆成两个小矩阵：

```text
y = W x + ΔW x
ΔW = B A
```

其中：

```text
A ∈ R^(r × d)
B ∈ R^(k × r)
```

这里最关键的是 `r`，也就是 **rank**。

rank 可以理解成 "真正独立的方向数量"。如果一个二维平面里有很多箭头，但它们都只是沿同一条直线伸缩，那么本质上只有 1 个独立方向，rank=1。如果它们能在横向和纵向两个方向上独立变化，rank=2。

放回 LoRA 里，`r=16` 的意思是：这个改变量不是在 `4096` 个方向上自由乱动，而是先把输入压到 16 个可训练的方向里，再从这 16 个方向展开回输出空间。

```text
x: 4096 维
↓ A：先压到 16 个方向
↓ B：再展开回 4096 维
得到一个小的修正量，加到原来的 Wx 上
```

所以如果 `d = 4096`、`k = 4096`、`r = 16`，LoRA 只需要训练：

```text
A: 16 × 4096
B: 4096 × 16
总计：16 × 4096 + 4096 × 16 = 131,072
```

约 13 万参数，而不是 1677 万参数。它不是训练一整台新摄像机，而是训练一个只有 16 个主调节方向的小镜头附件。

那它怎么影响 Attention？

如果 LoRA 加在 `W_Q` 上，它改变的是 token "想问什么"：

```text
原来：这个 token 用原来的 Q 去找相关信息
LoRA：在 Q 上加一点偏移，让它更会问当前任务需要的问题
```

如果 LoRA 加在 `W_K` 上，它改变的是 token "怎样被别人搜到"；加在 `W_V` 上，它改变的是 token "被选中后贡献什么内容"；加在 `W_O` 上，它改变的是多个 head 开完会之后，信息怎样被投影回主干表示。

所以 LoRA 常常优先加在 Attention 的 q/k/v/o projection 上。因为很多 SFT 任务不是让模型重新学习世界知识，而是让它改变信息路由方式：用户问话时该关注哪里，工具调用时该抽取哪些字段，回答时该沿着什么格式组织信息。

LoRA 还有几个常见超参数：

- `rank`：低秩空间的维度，越大容量越强，显存也越高；
- `alpha`：控制 LoRA 增量的强度，常见形式是 `W' = W + (alpha / r) BA`；
- `target_modules`：决定把 LoRA 加到哪些线性层上，常见选择是 Attention 里的 q/k/v/o projection。

推理时，LoRA 可以保持为一个外挂小模块，也可以直接 merge 回原权重：

```text
W' = W + B A
```

合并之后，模型推理时就不需要再额外挂一个 LoRA 分支，结构几乎和普通模型一样。

不过 nanoChat 这里没有实现 LoRA。它的目标是展示一条最直接的端到端训练链路：预训练出 base model，再把同一个模型继续 SFT 成 chat model。所以这里选择全参数 SFT，而不是参数更省、但工程结构更复杂的 LoRA/adapter 方案。

### 对话格式：给模型戴上角色名牌

普通文本里没有天然的 "用户" 和 "助手"。所以 nanoChat 会用 special tokens 把对话结构标出来：

```text
<|bos|>
<|user_start|>
请解释一下什么是 Transformer
<|user_end|>
<|assistant_start|>
Transformer 是一种基于注意力机制的神经网络架构……
<|assistant_end|>
```

这些 special tokens 就像剧本里的角色名牌：

```text
用户登场
用户说完
助手登场
助手说完
```

模型原本只是在学习 "下一个 token 是什么"。有了这些标记之后，它开始学会更具体的模式：

```text
看到 <|user_start|> 后面的是用户问题
看到 <|assistant_start|> 后面应该开始回答
看到 <|assistant_end|> 表示回答结束
```

这就是为什么聊天模型需要自己的对话模板。模板不是装饰，而是在告诉模型：这段文本里谁在说话，什么时候轮到你说。

### SFT 的 loss 不是所有 token 都算

预训练时，几乎每个 token 都是学习目标。因为模型的任务就是学整段文本的统计规律。

但 SFT 不一样。对话里有两类内容：

```text
用户说的话：上下文条件
助手说的话：模型要学习的目标
```

用户问题本身不是模型要模仿生成的答案。我们不希望模型学成 "用户问一句，它也跟着生成用户问题"。我们希望它读懂用户内容，然后生成助手回复。

所以 SFT 通常只对 assistant 的输出部分计算 loss。

可以把它想成老师批改角色扮演练习：

```text
用户台词：题目，不扣分
助手台词：学生答案，要评分
```

比如：

```text
<|user_start|> 1+1 等于几？ <|user_end|>
<|assistant_start|> 1+1 等于 2。 <|assistant_end|>
```

模型会看到完整上下文，但真正被评分的是：

```text
1+1 等于 2。 <|assistant_end|>
```

这背后通常用一个 mask 控制：

```text
mask = 0：只是上下文，不算 loss
mask = 1：助手输出，要算 loss
```

这个细节非常重要。它决定了模型到底在模仿什么。

### SFT 数据混合：不只学聊天语气

如果 SFT 数据只有普通闲聊，模型会变得更像聊天助手，但不一定会做题、算数、遵守格式。

所以 SFT 阶段会混合多种任务数据，例如：

- 普通对话数据：学会自然回答；
- 身份对话数据：学会自己是谁；
- MMLU：学会选择题格式和知识问答；
- GSM8K：学会小学数学推理；
- 拼写和计数任务：学会更精细的字符级能力。

这像训练一个新员工。不能只教他说 "您好，请问有什么可以帮到您"，还要教他查资料、做基础计算、看懂选择题、处理边界问题。

SFT 的本质不是给模型灌入全部知识。大部分知识已经来自预训练。SFT 更像把知识重新组织成可用的对话行为：

```text
你知道很多东西
现在请按助手的方式把它说出来
```

### SFT 之后，模型才像一个聊天机器人

经过 SFT 后，模型会更稳定地遵守这种结构：

```text
用户提问 → 助手回答 → 回答结束
```

它不再只是漫无目的地续写网页文本，而是更像一个会接话的助手。

但 SFT 仍然有局限。

它学的是 "人类示范答案长什么样"。如果某些问题需要精确计算、搜索、执行代码，模型单靠参数里的记忆和推理仍然可能出错。


## Tool Use：让模型把一部分事情交给工具

语言模型很擅长生成文字，但它不是计算器，也不是 Python 解释器。

比如问：

```text
23847 * 593 等于多少？
```

模型可以凭模式猜一个看起来像答案的数字，但这不是可靠计算。它在做的是语言预测，不是严格执行乘法算法。

所以更聪明的方式是：让模型在需要精确计算时调用工具。

这就像一个人写报告时可以打开计算器。我们并不会因为他用了计算器就说他不会思考。恰恰相反，知道什么时候该调用工具，本身就是能力的一部分。

### 工具调用也是一种特殊对话格式

GPT 用 special tokens 表示 Python 工具调用：

```text
<|python_start|>
23847 * 593
<|python_end|>
<|output_start|>
14141271
<|output_end|>
```

真实输出当然应该由工具计算出来。这里的结构重点是：

```text
assistant 写出要执行的表达式
系统执行表达式
执行结果被插回上下文
assistant 继续根据结果回答
```

这像一场开卷考试：

```text
学生：这题需要算一下，我写出算式
计算器：给出结果
学生：根据结果组织最终答案
```

模型不是直接凭空背出所有答案，而是学会把某些中间步骤交给更可靠的外部系统。

### 工具输出不是模型自己生成的

这里有一个很容易忽略的点：工具输出不应该被当成 assistant 要学习生成的内容。

因为工具输出来自执行环境，不来自模型本身。

在训练时，assistant 写出的 Python 表达式可以算作模型输出，要监督；但 Python 执行结果更像外部世界返回的信息，应该作为上下文喂回模型，而不是要求模型背下来。

可以类比成做实验：

```text
实验方案：学生写的，要评分
仪器读数：仪器给的，不要求学生凭空生成
实验结论：学生根据读数写的，要评分
```

这能帮助模型学到一种更健康的行为：

```text
我可以提出工具调用
我等待工具返回结果
我再基于结果继续回答
```

### Tool Use 让模型从 "会说" 变成 "会借力"

工具调用的意义不只是算乘法。

而是把语言模型从一个封闭的文本生成器，变成一个可以和外部系统协作的代理：

```text
语言模型：负责理解意图、拆解问题、组织答案
工具系统：负责精确计算、执行代码、查询状态
```

**模型不必把所有能力都塞进参数里。它可以学会在合适的时候调用工具。**



## 强化学习：让模型从自己的答案里学偏好

SFT 让模型学会模仿示范答案。但模仿有一个问题：

```text
示范数据告诉模型“人类答案长什么样”
但不一定直接告诉模型“多个答案里哪个更好”
```

比如一道数学题，模型可以生成 16 个不同答案：

```text
答案 A：推理正确，结果正确
答案 B：过程像真的，但最后算错
答案 C：直接猜了一个数字
答案 D：格式很好，但没回答问题
...
```

SFT 更像 "看标准答案学习"。强化学习则更像 "自己做题，然后根据得分调整策略"。

### Reward：给答案打分

强化学习里最核心的概念是 **reward（奖励）**。

对于某些任务，reward 可以很明确。比如 GSM8K 数学题，最后答案对了就是高分，错了就是低分。

这像做练习题：

```text
最后答案正确 → 加分
最后答案错误 → 不加分或扣分
```

模型先根据题目生成多个回答，然后系统检查这些回答是否正确。这个检查结果就变成训练信号。

注意，这和 SFT 的监督不一样。SFT 是直接告诉模型：

```text
标准答案应该这样写。
```

强化学习更像只告诉模型：

```text
你刚才这个回答得分高。
你刚才那个回答得分低。
```

它不一定逐字告诉模型应该怎么写，而是通过奖励引导模型多生成高分答案。

### 多采样：同一道题让模型答很多遍

RL 阶段会对同一个问题采样多个答案。

这很像让学生同一道题尝试多种解法：

```text
第 1 种解法：错在列式
第 2 种解法：过程对，算术错
第 3 种解法：过程和答案都对
第 4 种解法：绕远了，但最后对
```

如果我们只看一个答案，很难知道模型只是运气好还是策略真的好。多个答案放在一起，就能比较：哪些生成路径更值得鼓励，哪些路径应该少走。

### Advantage：比平均水平好多少

强化学习里常见一个概念叫 **advantage**，可以粗略理解成：

```text
这个答案比同组平均水平好多少？
```

假设同一道题采样了 4 个答案，得分是：

```text
[1, 0, 0, 1]
```

平均分是 0.5。那么得分 1 的答案就比平均好，得分 0 的答案就比平均差。

训练时，模型会被推向更常生成那些高于平均的回答，远离低于平均的回答。

这比只看绝对分数更稳一点。因为不同题目难度不同：

- 有的题很简单，大家都答对；
- 有的题很难，大家都答错；
- 真正有价值的是，在同一个问题里，哪些回答相对更好。

### RL 不是凭空变聪明，而是在调行为分布

强化学习听起来很神秘，但在这里可以先抓住一个朴素直觉：

```text
模型已经会生成很多种答案
RL 让它更偏向生成得分高的那几种
```

它不是突然给模型灌入一本数学教材。预训练和 SFT 已经提供了大量知识和基本行为。RL 更像最后的偏好调整：让模型在多个可能回答之间，更愿意走向被奖励的路径。

比如数学题里，它可能会更愿意：

- 多写一步推理；
- 最后明确给出答案；
- 使用工具检查计算；
- 少给没有依据的猜测。

所以强化学习可以理解成：

```text
预训练：学会语言和知识
SFT：学会像助手一样回答
RL：学会更偏向高分回答
```



## 推理：训练好的模型怎样一句句说话

训练阶段，模型一次看到整段文本，并在每个位置同时预测下一个 token。

推理阶段就不一样了。

用户只给一个 prompt，模型要真的开始往后写。它每次只能生成一个 token：

```text
已有文本 → 预测下一个 token → 拼回文本末尾 → 再预测下一个 token → ...
```

这就是 **自回归生成（autoregressive generation）**。

### 生成不是一次写完整篇，而是一格一格往前走

假设用户问：

```text
什么是机器学习？
```

模型内部不是一下子吐出整段回答，而是像打字一样一步步生成：

```text
机器
机器学习
机器学习是
机器学习是一种
机器学习是一种让
...
```

每一步，它都会重新问自己：

```text
在当前已有内容后面，下一个 token 最可能是什么？
```

这和训练目标完全一致。训练时练的就是预测下一个 token，推理时用的也是预测下一个 token。区别只是：

```text
训练时：真实答案已经在数据里，用来算 loss
推理时：真实答案不存在，模型自己的输出会变成下一步上下文
```

这也是为什么早期错误会影响后续生成。模型一旦生成了某个 token，后面就要沿着这个 token 继续写，就像写文章时前一句话已经落笔，后面只能接着它往下圆。

### Temperature：回答要稳，还是要发散

模型每一步都会给整个词表一个概率分布。

如果总是选择概率最高的 token，叫 **贪心生成**。它很稳定，但有时会无聊、重复、缺少创造性。

如果按概率随机采样，回答会更丰富，但也更容易跑偏。

**temperature** 就是控制随机性的旋钮。

我们可以把 logits 想成一张候选名单：

```text
候选 A：很可能
候选 B：也可以
候选 C：有点冒险
候选 D：非常离谱
```

低 temperature 会让高分候选更突出，模型更保守：

```text
适合事实问答、代码、数学
```

高 temperature 会把分布压平，让低分候选也有机会出现，模型更发散：

```text
适合头脑风暴、故事、诗歌
```

它像调味：太淡没变化，太重会盖住原味。

### Top-k：只从前几个靠谱候选里抽

除了 temperature，还有一种常见办法叫 **top-k sampling**。

它的想法很简单：

```text
只保留概率最高的 k 个 token
其他候选直接丢掉
再在这 k 个里面采样
```

这像考试做选择题时先排除明显不靠谱的选项，再在剩下几个里选。

top-k 可以减少模型抽到离谱 token 的概率，同时保留一定随机性。

### KV Cache：别每次都从头读一遍

推理时还有一个巨大的效率问题。

假设模型已经生成了 1000 个 token，现在要生成第 1001 个。最朴素的做法是：把前 1000 个 token 全部重新送进模型，再算一次 Attention。

但这很浪费。因为前 999 个 token 的 key/value 其实早就算过了。

KV Cache 的想法就是：

```text
每生成一个 token，就把它对应的 K/V 存起来
下次生成时，旧 K/V 直接复用
只为新 token 计算新的 K/V
```

可以类比读长篇小说。

没有缓存时，每写下一句评论，你都要从第一页重新读到当前页。

有缓存时，你已经做过读书笔记。下一次只要翻看笔记，再读新出现的一行，就能继续判断。

KV Cache 不改变模型答案的数学含义，它只是避免重复计算，让生成速度快很多。

### GQA：让缓存更省空间

前面讲多头注意力时说过，每个 head 都会产生自己的 K/V。推理时这些 K/V 都要缓存，上下文越长，占用显存越大。

GQA（Grouped Query Attention）的思路是：

```text
多个 Query head 共享一组 Key/Value head
```

这像一个团队里有很多提问者，但他们共用一套资料库。每个人提问角度不同，但不一定每个人都要维护一份完整档案。

这样做能显著减少 KV Cache 的体积，让长上下文推理更省显存。

### 从模型到聊天界面

模型本身只认识 token id。它不会天然知道网页按钮、输入框、聊天气泡是什么。

所以真正的 ChatGPT 风格界面还需要一层包装：

```text
用户在网页输入文字
→ Tokenizer 编码成 token
→ 按聊天模板加上 user/assistant 标记
→ 模型逐 token 生成
→ Tokenizer 解码回文字
→ 前端显示成聊天气泡
```

命令行聊天和 GUI 本质上都在做这件事。区别只是外壳不同：一个在终端里显示，一个在浏览器里显示。


## 实机演示：chat model 已经像助手了

前面我们已经看过 base model：它会续写，但不会稳定地扮演助手。现在再看 SFT 之后的 chat model：

```bash
bash runs/chat_d34_sft_cli.sh
```

现在我们输入同一个问题：

```text
User: why the sky is blue?

Assistant: The sky appears blue to us because of a phenomenon called Rayleigh scattering...
```

这时模型已经明显更像助手了。它会识别 `User:` 的提问，进入 `Assistant:` 的回答角色，并尝试组织一段完整解释。也就是说，SFT 没有把模型从零变聪明，而是把预训练里学到的语言和知识，重新整理成一种可交互的行为。

但这个演示也暴露了另一个事实：**会聊天，不等于一定可靠。**

比如继续追问：

```text
User: tell me about Rayleigh scattering formula

Assistant: The formula for Rayleigh scattering is given by:

R = (4π) / λ
```

这个公式就是错的。Rayleigh scattering 最关键的波长关系不是 `1 / λ`，而是近似：

```text
scattering intensity ∝ 1 / λ^4
```

这正好说明了 base → SFT 的提升边界：

- base model 学会了续写世界；
- SFT 让它学会了像助手一样回答；
- 但事实准确性、数学推理、公式校验、工具使用、拒绝胡编，还需要更强的数据、训练和系统设计。

所以从一个小型 ChatGPT 到 GPT-4、GPT-5 这类更强系统，关键提升并不是某一个神秘开关，而是一组东西叠加起来。

### 能力提升的关键

第一是 **规模**。

更大的模型、更长的训练、更大的 token 预算，让模型能吸收更多语言模式、知识结构和推理模式。规模不是万能的，但它确实提供了更大的容量。很多能力不是单条规则写进去的，而是在足够多的数据和足够大的模型里逐渐涌现出来的。

第二是 **数据质量**。

模型最终学到什么，很大程度上取决于它吃过什么。重复、低质、错误、互相矛盾的数据会让模型学到噪声；高质量教材、代码、数学、科学问答、真实对话、严谨推理过程，则会把模型往更可靠的方向推。到了大模型阶段，数据清洗、去重、配比和课程安排，往往不比模型结构本身次要。

第三是 **后训练**。

预训练得到的是 base model，SFT 把它训练成助手，RL / DPO / 偏好优化再进一步告诉它：什么回答更好，什么回答虽然流畅但不该给。很多 "更有用、更听话、更少胡说" 的体验，来自后训练，而不是单纯来自预训练 loss 更低。

第四是 **可验证能力**。

公式、计算、代码、事实查询这些问题，不应该完全靠参数里的模糊记忆。更强的系统会让模型调用搜索、数据库、计算器、Python、代码执行器，或者至少在训练中学会更强的自检。Rayleigh scattering 公式这种错误，正是因为模型只是在语言上 "像一个解释"，但没有真正经过外部校验。

第五是 **推理时的系统工程**。

同一个模型，用不同的 prompt 模板、采样参数、上下文管理、工具协议、检索系统和安全策略，表现会差很多。ChatGPT 不是一个裸模型直接接在输入框后面，而是一整套围绕模型构建的产品系统。

```text
预训练给模型知识和语言直觉，SFT 给模型助手形态，RL/偏好优化给模型行为方向，工具和系统工程给模型可靠性边界。
```

到这里，ChatGPT 的整条路线就讲完了：先有一个会续写的 base model，再把它调成会接话的 chat model，然后我们也亲眼看到，下一步要追求的不是更会说，而是更可靠、更会验证、更会借助工具

```text
原始文本
→ 训练 Tokenizer
→ 预训练 base model
→ 用 BPB / CORE / sample 评估
→ SFT 成聊天模型
→ 学会必要的 Tool Use
→ 用 RL 偏向更高分答案
→ 用 KV Cache 高效推理
→ CLI / GUI 里真正对话
```

从外面看，聊天机器人像是 "输入一句话，输出一段话"。但拆开以后，它其实是一条很长的工程链路：

```text
文本切块
向量表示
上下文建模
概率预测
损失优化
行为微调
工具协作
采样生成
界面呈现
```

ChatGPT 的神奇感，就来自这条链路的叠加：每一步单独看都不算玄学，但当足够大的模型、足够多的数据、足够长的训练和足够细的工程设计堆在一起，它就从一个 "预测下一个 token" 的系统，变成了一个我们可以对话、提问、协作的小型智能体。
